# Hybrid Cloud Lakehouse
*An ELT Pipeline with AWS, Apache Iceberg, Snowflake, and dbt*

This project is a modern, automated data platform built from scratch using an ELT architecture. The goal was to integrate three different data sources, process them, and calculate,all while keeping cloud costs low and data strictly organized.

### Architecture Flow
The pipeline moves data from live generators to an analytical dashboard. Airflow acts as the central orchestrator, triggering extraction, load, and transformation jobs in sequence.
```text
[ PostgreSQL (PARQUET) ]
[ Python Script (JSON) ]  ----> [ Apache Airflow (Docker) ] ----> [ AWS S3 (Bronze Layer) ]
[ Exchange Rate API (CSV) ]                                          |
                                                                     | (External Tables & Partition Pruning)
                                                                     v
+--------------------------------------------------------------------+
|                          Snowflake (Compute)                       |
|                                                                    |
|  1. Bronze: External Tables reading S3                              |
|  2. Silver: dbt Incremental Iceberg tables (cleaned & typed)        |
|  3. Gold: dbt Incremental Iceberg tables (Star Schema & SCD Type 1) |
+--------------------------------------------------------------------+
                                                                     |
                                                                     | (Reads Iceberg tables via SQL)
                                                                     v
                                                          [ Metabase Dashboard ]
```

## Tech Stack
- **Terraform** - Manages all infrastructure as code. It provisions the AWS S3 buckets, IAM roles, and Snowflake databases, automating the secure connection between the two platforms without hardcoding keys.

- **Airflow & Docker** - The orchestrator. Runs in a Docker container and triggers the Python data generators, manages the S3 uploads, and automatically executes dbt transformations and tests.

- **S3** - The physical storage layer. Holds the raw landing data (Bronze) partitioned by date, as well as the final Apache Iceberg tables (Gold & Silver).

- **Snowflake** - The compute engine. Instead of storing data natively, it reads directly from S3 using External Tables and executes the heavy SQL transformations, writing the final results back to S3.

- **dbt** - Handles the T in ELT. It builds the progressive Silver and Gold layers, manages the incremental logic (MERGE), models the Star Schema, and runs data quality tests.

- **Apache Iceberg** - The open table format used for the Gold layer. It brings ACID transactions to S3, allowing Snowflake to write structured data while keeping it free from vendor lock-in.

- **GitHub Actions** - CI/CD pipeline. It runs `dbt compile` and `dbt test` to ensure broken code or bad data models never reach the main branch.

- **Metabase** - The visualization layer. It connects directly to Snowflake to query the Iceberg tables and display business metrics, like global revenue converted to USD.

## Infrastructure as Code & Security
Instead of clicking through the AWS and Snowflake consoles, the entire cloud setup is defined as code using Terraform. This ensures the environment is fully reproducible and version-controlled.

* **Key Pair Authentication** - To connect tools like dbt and Airflow to Snowflake, I used RSA key pairs instead of standard passwords. This is a more secure, machine-to-machine authentication method that avoids hardcoding credentials in environment files.
```terraform
provider "snowflake" {
  role              = "ACCOUNTADMIN"
  private_key_path  = "${path.module}/../ecom_dbt/snowflake_key.p8"
}
```

* **IAM Roles & External ID** (Confused Deputy Prevention) - Snowflake does not use AWS access keys to read from S3. Instead, I configured an AWS IAM Role that Snowflake assumes using the STS AssumeRole action. To prevent the "Confused Deputy" attack vector, the role's Trust Policy strictly checks a secret External ID generated by Snowflake.
```hcl
principals {
      type        = "AWS"
      identifiers = [snowflake_storage_integration.s3_integration.storage_aws_iam_user_arn, snowflake_external_volume.gold_iceberg_volume.describe_output[0].storage_locations[0].s3_storage_location[0].storage_aws_iam_user_arn]
    }
    condition {
      test     = "StringEquals"
      variable = "sts:ExternalId"
      values = [coalesce(
        snowflake_storage_integration.s3_integration.storage_aws_external_id,
        snowflake_storage_integration.s3_integration.describe_output[0].storage_aws_external_id[0].value,
        "dummy_id"), coalesce(
        snowflake_external_volume.gold_iceberg_volume.describe_output[0].storage_locations[0].s3_storage_location[0].storage_aws_external_id,
        "dummy_id_2")
      ]
    }
```
*(Note: `coalesce` and `describe_output` logic shown above was a necessary workaround for a specific provider bug during the Terraform plan phase. I explain exactly how and why I built this in the "Key Challenges" section later.)*
* **Overcoming Terraform Circular Dependency** - Creating the Snowflake Storage Integration required an AWS IAM Role ARN, but creating the IAM Role required the Snowflake IAM User ARN. To break this chicken-and-egg cycle in a single `terraform apply`, I dynamically constructed the IAM Role ARN using the AWS Account ID data source. This allowed Snowflake to create the integration first, pass its generated user ARN back to Terraform, and then apply the Trust Policy to AWS.
```hcl
resource "snowflake_storage_integration" "s3_integration" {
  storage_aws_role_arn = "arn:aws:iam::${data.aws_caller_identity.current.account_id}:role/snowflake_s3_read_role"
}
```

## Data Lakehouse & Orchestration
Apache Airflow runs in Docker and acts as the central orchestrator. Every day, it triggers Python scripts that generate mock e-commerce orders (PostgreSQL), user clickstream logs (JSON), and fetch live exchange rates (CSV API).

* **Hive-style Partitioning on S3** - When uploading the extracted files to AWS S3, the data is strictly organized into folder paths using the `event_date=YYYY-MM-DD` convention. This is a must have step for cloud cost optimization.
```python
s3_clickstream = f"bronze/clickstream/event_date={execution_date}/clickstream.json"
s3_client.upload_file(local_clickstream, bucket_name, s3_clickstream)
```

* ***Snowflake External Tables*** - Instead of copying raw files into Snowflake's internal storage, I defined External Tables. Snowflake simply reads the files directly from S3, meaning storage costs are kept to an absolute minimum. I managed these tables using the `dbt_external_tables` package, defining them as standard `sources` in dbt.

```yaml
sources:
  - name: bronze
    database: ECOM_DB
    schema: BRONZE
    tables:
      - name: ext_core_system
        external:
          location: "@ECOM_DB.BRONZE.S3_CORE_SYSTEM_STAGE"
          file_format: "( FORMAT_NAME = ECOM_DB.BRONZE.PARQUET_FORMAT )"
          partitions:
            - name: event_date
              data_type: date
              expression: "to_date(substr(split_part(metadata$filename, 'event_date=', 2), 1, 10), 'YYYY-MM-DD')"
```

* ***Partition Pruning*** - By explicitly defining the `event_date` partition in the YAML above, Snowflake knows exactly which S3 folders to look in. When a query asks for data from a specific date, Snowflake skips all other folders entirely. This drastically reduces the bytes scanned and speeds up query execution.

## Transformations & Modeling
This is where dbt handles the 'T' in ELT. Instead of Snowflake storing data natively, dbt orchestrates SQL to read from the Bronze External Tables, clean and structure the data, and write it back to S3 as physical Apache Iceberg tables.

The final Gold layer is structured as a Star Schema to optimize analytical queries.

link

* ***Incremental Iceberg Tables*** - To avoid scanning and rewriting the entire history on every pipeline run, the Silver and Gold layers are built as incremental models. dbt uses a "High Water Mark" strategy. It queries the target Iceberg table for the maximum `event_date`, and then asks the Bronze source only for partitions newer than that date. It then uses the `MERGE` strategy to append or update the new records.
```sql
{% if is_incremental() %}
    WHERE o.event_date > (SELECT MAX(event_date) FROM {{ this }})
{% endif %}
```

* ***Asynchronous Star Schema & Surrogate Keys*** - To speed up dbt run times, fact and dimension tables are built without physical `JOINs` during the ETL phase. Instead, I generate MD5 Surrogate Keys in-flight. Because the `generate_surrogate_key` macro is deterministic, applying it to `customer_id` in `fact_orders` and `dim_customer` produces the exact same hash. They will join perfectly in the BI tool without requiring a dependency in the dbt DAG.
```sql
-- Inside fact_orders.sql
{{ dbt_utils.generate_surrogate_key(['o.customer_id']) }} AS customer_sk

-- Inside dim_customer.sql
{{ dbt_utils.generate_surrogate_key(['customer_id']) }} AS customer_sk
```
* ***SCD Type 1 for Behavioral Tracking*** - The `dim_customer` table tracks whether a user has ever placed an order or just clicked around. Because it's an incremental model, a simple `MERGE` would overwrite a user's "has_ordered" flag to `FALSE` on a day they only browsed. To prevent losing this historical state, the model uses a `LEFT JOIN` against its own target table (`{{ this }}`), combining today's activity with the historical flags using an `OR` `COALESCE` logic.
```sql
SELECT 
    tc.customer_id,
    -- If they ordered today OR ordered in the past, keep it TRUE
    (tc.has_orders_today OR COALESCE(dc.has_orders, FALSE)) AS has_orders
FROM today_customers tc
LEFT JOIN {{ this }} dc ON tc.customer_id = dc.customer_id
```

## CI/CD & Data Quality
In a team environment, you can't just push code directly to production and hope it doesn't break the warehouse. This project implements a CI/CD pipeline using GitHub Actions to enforce data quality and code safety before any changes are merged.

* ***Automated CI Pipeline*** - Every time a Pull Request is opened against the main branch, GitHub Actions spins up a fresh environment, installs dbt, and runs validation checks. Instead of running a full `dbt run` (which would consume Snowflake credits and modify tables), the pipeline runs `dbt compile` to validate SQL syntax and Jinja logic, followed by `dbt test`.
```yml
- name: Compile dbt models
  run: |
    cd ecom_dbt
    dbt compile

- name: Run dbt tests
  run: |
    cd ecom_dbt
    dbt test
```

* `Secure Configuration` - The CI/CD pipeline needs to authenticate with Snowflake. Connection details and the RSA private key are stored as GitHub Secrets. They are injected at runtime as environment variables, which dbt reads dynamically using the env_var() function in `profiles.yml`.
```yaml
# profiles.yml
account: "{{ env_var('SNOWFLAKE_ACCOUNT') }}"
user: "{{ env_var('SNOWFLAKE_USER') }}"
private_key_path: "snowflake_key.p8"
```

* ***Data Contracts*** - The Gold layer is strictly tested. Using `schema.yml`, I defined data contracts that act as tests. It checks for `unique` and `not_null` values on primary keys. More importantly, it checks `relationships` to ensure Referential Integrity - guaranteeing that every `customer_sk` in the `fact_orders` table actually exists in `dim_customer`, preventing orphaned facts.
```yaml
models:
  - name: fact_orders
    columns:
      - name: customer_sk
        tests:
          - not_null
          - relationships:
              to: ref('dim_customer')
              field: customer_sk
```

## Summary and Key Learnings
Building a hybrid cloud Lakehouse from scratch was a deep dive into modern data engineering. It wasn't without its roadblocks, but solving them was the most rewarding part of the project. Here are the main challenges and what I took away from them.

### Key Challenges

1. ***The Terraform Provider Bug***

Snowflake Terraform provider had a bug where the `storage_aws_external_id`attribute returned `null` during the terraform plan phase. Since AWS Trust Policies reject null values, the plan kept failing.
To fix this, I dug into the Terraform state file (`tfstate`) and discovered the real ID was hidden inside a nested `describe_output` block. I used the `coalesce()` function to check the primary attribute first, fall back to the `describe_output` value if it existed, and finally use a `dummy_id` just to let the plan generate. During the `apply` phase, Terraform fetched the real ID and applied the correct Trust Policy to AWS. yipee!

2. ***Circular Dependencies***

Connecting Snowflake to AWS S3 securely required creating a Snowflake Storage Integration and an AWS IAM Role. However, this created a classic chicken-and-egg problem: *Snowflake needed the AWS IAM Role ARN to create the integration, but the AWS IAM Role needed the Snowflake IAM User ARN to set up its Trust Policy.* Terraform threw a cycle error because neither resource could be created first.
To break this cycle, I dynamically constructed the AWS IAM Role ARN as a string using the AWS Account ID data source (`data.aws_caller_identity.current.account_id`). Since Snowflake just needed a string to start the integration, Terraform created the integration first, extracted the generated IAM User ARN, and then created the AWS Role.

3. ***Refactoring the Pipeline Architecture***

My initial design for the Silver layer relied on Snowflake views reading directly from S3 stages. While simple to write, I realized this meant every analytical query would have to parse all raw files on the fly. To fix this, I completely refactored the architecture mid-project. First, I introduced partitioned External Tables in the Bronze layer (using the `dbt_external_tables` package), explicitly defining `event_date` partitions from the S3 folder paths. This enabled Snowflake's Partition Pruning at the storage level. Then, I changed the Silver and Gold layers to physical Apache Iceberg tables with incremental materialization. By combining the partitioned External Tables with dbt's "High Water Mark" pattern, the pipeline forces Snowflake to physically skip old S3 folders during dbt runs, scanning only new partitions and writing the refined results back to S3 as Iceberg. This created a cascading partition pruning effect across the whole architecture.

4. ***Preserving Historical State in Incremental Dimensions***

When building `dim_customer` as an incremental model using the `MERGE` strategy, I hit a logical trap. If a customer placed an order yesterday (`has_orders = TRUE`) but only browsed the site today, the incremental run would try to update their record with today's data, potentially wiping the has_orders flag to `FALSE`. To solve this without using complex SCD Type 2 history tracking, I implemented a self-join against the target table (`LEFT JOIN {{ this }}`). By combining today's activity with the historical flags using an `OR COALESCE` logic, the model safely updates the dimension without losing past behavioral data.

### Key Takeaways

* ***Infrastructure is Code, but Code can be moody*** - Relying on providers (like Snowflake's Terraform provider) means you have to be prepared for bugs in their code. Reading the tfstate file and understanding the tool's internal data structures is sometimes the only way forward.

* ***Compute can be Cheap if You Respect Partitions*** - The biggest "aha" moment was seeing the Snowflake Query Profile in action. By layering Hive-style partitions on S3, External Tables, and Iceberg `partition_by`, the Partition Pruning cascaded through the architecture. Snowflake scanned kilobytes instead of gigabytes, proving that good architecture is the best FinOps.

* ***Modeling isn't just writing SELECTs*** - Building a Star Schema isn't just about joining tables. Understanding the grain of a dimension, implementing SCD Type 1 logic safely, and generating Surrogate Keys asynchronously to break dbt dependencies completely changed how I think about ETL performance.

## Before You Run
To spin up this environment from scratch, you need an AWS account, a Snowflake account, and Docker installed on your machine.

### Precondition
1. Create two .`env` files based on the templates provided at the bottom. Place one in the `terraform/` directory and the other in the project root folder (where `docker-compose.yml` is).

2. Generate an encrypted RSA key pair for Snowflake authentication. Place the private key (`snowflake_key.p8`) inside the `ecom_dbt/` folder. You will be prompted to create a passphrase - remember it, as you will need it for the .env file.

3. Assign the public key to your Snowflake user. Log into your Snowflake console (using the ACCOUNTADMIN role) and run the following SQL, replacing the value with the contents of your `snowflake_key.pub` file (without the -----BEGIN/END----- headers):
```sql
ALTER USER your_snowflake_username SET RSA_PUBLIC_KEY='MIIBIjANBgkqhkiG9w0B...';
```
---

### Execution

1. ***Initialize and Create Infrastructure***

Navigate to the terraform/ directory, load your environment variables, and deploy the AWS/Snowflake infrastructure.
```bash
cd terraform/
source .env
terraform init
terraform apply
```

2. ***Launch Docker Containers***

Return to the project root and start Airflow and Metabase.
```bash
cd ..
docker compose up -d --build
```

3. ***Bootstrap the Static Dimension***

Run this command in your terminal to execute dbt inside the Airflow container. This creates the calendar table in Snowflake once (it is excluded from the daily Airflow DAG to save compute).
```bash 
docker compose exec webserver bash -c "cd /opt/airflow/ecom_dbt && dbt run --select dim_date"
```

4. ***Trigger the Pipeline (Optional)***

Open the Airflow UI at http://localhost:8080 (login: admin / admin) and trigger the ecom_data_pipeline DAG. It will generate data, upload to S3, build the Iceberg tables, and run tests. Once finished, check the dashboard at http://localhost:3000.
